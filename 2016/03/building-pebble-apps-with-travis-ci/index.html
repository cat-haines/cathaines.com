Building Pebble Apps with Travis CI

Filed under: code, pebble, and travis | Mar 11, 2016

I‚Äôve been spending a fair bit of time tinkering with Travis CI lately, and it‚Äôs probably pretty safe to say that it‚Äôs one of my favourite tools at the moment. Travis CI, for those unfamiliar with it, is a continuous integration tool, it enables developers to easily build, test, and deploy their projects on an ongoing basis.

In this post, we‚Äôre going to look at how to setup Travis CI to build a Pebble application.

tl;dr

If you‚Äôre looking to get up and running as quickly as possible, you can follow these simple steps:

  • Fork the pebble-travis project.
  • Create a Travis CI account if you don‚Äôt already have one.
  • Find the repository you just forked on your Travis CI profile page.
  • Toggle the build switch for the repository.

And that‚Äôs really all that‚Äôs required. Whenever you push a change to the cloned repository, it will kick off a new Travis CI build!!

The rest of this blog post will look at how the actually integration works, and hopefully provide enough information to enable to you modify and extend the project to suite you specific needs.

Before We Get Started

In order to follow along with this blog post, you‚Äôre going to need a few things:

  • A GitHub account
  • A Travis CI account
  • A Pebble project hosted on GitHub

We will also need to enable the Travis integration for the GitHub repo we want to build. Don‚Äôt worry - Travis and GitHub have a tight integration, and this step is almost non-existent: head over to your profile page and ‚ÄúFlick the repository switch on.‚Äù

.travis.yml

The next step is to create a .travis.yml file, which specifies what Travis CI will do when we push to GitHub. Here‚Äôs what our base file is going to look like:

sudo: false

language: python
python:
 - 2.7

We‚Äôve specified sudo: false, which tells Travis CI to build our project in a container-based (Linux) environment. Using a container-based environment ensures that our builds will spin up as quickly as possible after a commit has been pushed, but we won‚Äôt be allowed to use sudo in any of our scripts..

We‚Äôve also specified language: python. This may seem a bit odd at first glance, but since the Pebble Tool is written in Python, this will make installing the tool (and its dependencies) a lot simpler!

The Build Lifecycle

So what actually happens when a build is triggered in Travis CI?

The first thing Travis does is look at the .travis.yml file to determine what environment and language the developer has selected. The combination of the environment and language determines what set of tools and resources come pre-installed in our build environment.

Once the base environment has been setup, the build lifecycle begins. Each step in the build lifecycle has a default command that can be overridden by specifying new behaviour in our config file. We‚Äôre going to add the following to the end of our .travis.yml file:

before_install:
 - echo before_install
install:
 - echo install

before_script:
 - echo before_script
script:
 - echo script

If we commit and push these changes, the resulting build should pass and include the following logs:

$ python --version
Python 2.7.9
$ pip --version
pip 6.0.7 from /home/travis/virtualenv/python2.7.9/lib/python2.7/site-packages (python 2.7)
$ echo before_install
$ echo install
$ echo before_script
$ echo script

Now that we understand the basics of the build lifecycle, and the order in which scripts execute, we can prepare our project for a slightly more complex example. We‚Äôre going to create some custom bash scripts to be executed during the install and script steps, and use the before_install and before_script steps to grant them execute privileges:

# Grant the install script execute privileges
before_install:
 - chmod +x ./scripts/ciinstall

# Run a script to install our build chain
install:
 - ./scripts/ciinstall

# Grant the build script execute privileges
before_script:
 - chmod +x ./scripts/cibuild

 # Run a script to build and test our project
script:
 - ./scripts/cibuild

The last thing we need to do to get this new build to work, is create /scripts/ciinstall and /scripts/cibuild. Here‚Äôs what they‚Äôre going to look like:

#!/usr/bin/env bash
# {project_root}/scripts/ciinstall
set -e # halt script on error

echo Hello Install Step

#!/usr/bin/env bash
# {project_root}/scripts/cibuild
set -e # halt script on error

echo Hello Script Step

Adding these new files (along with the changes to our .travis.yml), and pushing them to GitHub should result in a successful build:

Python 2.7.9
$ pip --version
pip 6.0.7 from /home/travis/virtualenv/python2.7.9/lib/python2.7/site-packages (python 2.7)
before_install
$ chmod +x ./scripts/ciinstall
$ ./scripts/ciinstall
$ chmod +x ./scripts/cibuild
$ ./scripts/cibuild

Done. Your build exited with 0.

Installing the Pebble Tool

We‚Äôre in great shape at this point. We could take our .travis.yml file along with our /scripts folder and use it as a starting point for just any project we want to build in Travis (we could just need to change the language if we were working with something other than Python).

Let‚Äôs take a look at how we can extend our current Travis setup to install the Pebble Tool, and build our project!

Download and install the Pebble Tool

First things first - we need to download and install the latest version of the Pebble Tool. To do this, we‚Äôll add the following to the bottom of our ciinstall script:

mkdir pebble-dev
cd pebble-dev
curl -O https://s3.amazonaws.com/assets.getpebble.com/pebble-tool/pebble-sdk-4.2-linux64.tar.bz2
tar -jxf pebble-sdk-4.2-linux64.tar.bz2

That was easy, but there‚Äôs one problem - we‚Äôll need to update our ciinstall script every time there‚Äôs an update to the Pebble Tool. We want to avoid modifying our install script once it‚Äôs working, so we‚Äôre going to define what version of the Pebble Tool it should fetch through an environmental variable. We can set the environmental variable by adding the following to the bottom of our .travis.yml file:

env:
  global:
    - PEBBLE_TOOL=4.2

This allows us to rewrite our ciinstall with $PEBBLE_TOOL in place of ‚Äú4.2‚Äù:

mkdir pebble-dev
cd pebble-dev
curl -O https://s3.amazonaws.com/assets.getpebble.com/pebble-tool/pebble-sdk-$PEBBLE_TOOL-linux64.tar.bz2
tar -jxf pebble-sdk-$PEBBLE_TOOL-linux64.tar.bz2

Download and Install Python Libraries

The next step in the install guide tells us to install Python 2.7, and pip. We‚Äôre able to skip this step since Python 2.7 and pip will come pre-installed in our Travis CI environment. We do however, still need to perform the virtualenv step, so we‚Äôll add the following to the bottom of ciinstall:

cd pebble-sdk-$PEBBLE_TOOL-linux64
virtualenv --no-site-packages .env
source .env/bin/activate
pip install -r requirements.txt
deactivate

Install Pebble Emulator Dependencies

Finally, we need to install the emulator‚Äôs dependencies with apt-get. The guide tells us to run sudo apt-get install libsdl1.2debian libfdt1 libpixman-1-0, which we can‚Äôt do, as we‚Äôre not allowed to use sudo.

Instead, we‚Äôre going to specify the packages in our configuration file and let Travis take care of installing them for us. We can do this by adding the following to the bottom of our .travis.yml file:

addons:
  apt:
    sources:
      - libsdl1.2debian
      - libfdt1
      - libpixman-1-0

Building a Pebble App

Now that our build environment has the latest version of the Pebble Tool installed, building out project is as simple as adding the following line to our cibuild script:

# Pipe 'yes' into pebble tool to accept terms and conditions
yes | ./pebble-dev/pebble-sdk-$PEBBLE_TOOL-linux64/bin/pebble build

This command will build your Pebble project (and automagically answer ‚Äòyes‚Äô when the Pebble Tool asks whether you accept the terms and conditions).
